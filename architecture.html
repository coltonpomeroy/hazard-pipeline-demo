hazpipe // architecture sketch

Staging raw weather data for visualization and workflows

One ingest of the raw model data on S3, split into two tracks: a fast visual path for smooth maps and animation, and a queryable path that business logic can act on. Decoupling render from query is what a visualization-only API can't give you.

fast visual path queryable data path core staging layer

Why two tracks

A visualization-only API (e.g. Aguacero's) hands you a layer to draw, not records you can query. The moment the job is act on the data — alert when gust > X over these assets, store every cycle to query history, join weather to a customer's locations — you need the data in a database, not a tile. One ingest feeds both: tiles/Zarr for buttery animation, PostGIS + time-series for the workflows.

Where the live demo sits

The demo is a working slice of the spine: it pulls real HRRR from NOAA, normalizes it, rechunks to Zarr, and turns the gust field into vector hazard zones that animate on a MapLibre map — already EPSG:4326 and PostGIS-ready. The next build-out is the tiling layer for high-res animation and the DB layer the workflows run on.
